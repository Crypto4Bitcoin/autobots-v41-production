# AutoBots v4 build order

This is the strict order to move the project from scaffold to operational system.

## Phase 1. Install and validate the shell
1. `npm install`
2. `npm run validate:divisions`
3. `npm run typecheck`
4. `npm run lint`
5. `npm run test`
6. `npm run build`

Exit condition:
- build passes
- route registry is clean
- no type failures

## Phase 2. Connect Supabase
1. Create Supabase project
2. Add env vars:
   - `NEXT_PUBLIC_SUPABASE_URL`
   - `NEXT_PUBLIC_SUPABASE_ANON_KEY`
   - `SUPABASE_SERVICE_ROLE_KEY`
3. Run `db/schema.sql`
4. Run `node ./scripts/seed-divisions.mjs`
5. Run `db/seeds/generated_divisions.sql`

Exit condition:
- divisions table seeded
- dashboard loader switches from mock to database mode

## Phase 3. Real data loading
1. Populate `sources`, `entities`, `events`, `watch_scores`, `metric_series`, `metric_cards`
2. Replace starter views with production views
3. Test each route with real rows

Exit condition:
- every division page renders from DB

## Phase 4. Action controls
1. Wire buttons to real automation jobs
2. Keep mock-safe fallback for dev
3. Implement:
   - refresh signals
   - run verifier
   - run simulation
   - resolve issue
   - approve or reject relay
   - create entity
   - log event

Exit condition:
- action buttons mutate data or enqueue jobs

## Phase 5. Verifier engine
1. Create verifier queue table consumers
2. Implement source confidence scoring
3. Add stale-data checks
4. Add contradiction checks
5. Add relay validation
6. Add issue creation from detector failures

Exit condition:
- integrity issues are generated by live checks

## Phase 6. Automation layer
1. Add scheduled jobs
2. Add fetchers for public sources
3. Add simulation scheduler
4. Add periodic integrity scan
5. Add relay job runner

Exit condition:
- jobs run without manual clicking

## Phase 7. Auth, tenants, billing
1. Enable Supabase Auth
2. Add tenant records
3. Add tenant-user membership
4. Add plan limits and billing provider
5. Add audit trail

Exit condition:
- multi-tenant usage is possible
- billing limits are enforceable

## Phase 8. Staging proof
1. Deploy staging
2. Prove DB read and write
3. Prove auth
4. Prove job runs
5. Prove failure recovery
6. Prove logs
7. Prove rollback

Exit condition:
- staging behaves like production

## Phase 9. Production completeness
1. Expand missing ingestion providers
2. Harden permissions
3. Add observability
4. Add backups
5. Add incident playbook

Exit condition:
- complete operational baseline
